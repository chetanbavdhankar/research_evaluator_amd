# Phase 1: Paper Comprehension and Detail Extraction

## Purpose

The goal of this phase is to produce a complete, structured, and queryable knowledge artifact from the input paper. This artifact serves as the single source of truth for all downstream phases. Nothing should be assumed, inferred, or invented beyond what is explicitly stated or clearly implied in the paper. Every downstream phase depends entirely on the quality and completeness of this phase's output.

The input to this phase is a PDF, HTML page, or arXiv link. The output is a structured document — hereafter called the **Paper Knowledge Artifact (PKA)** — that captures the paper's full content at varying levels of detail depending on reproducibility relevance.

---

## Core Principle: Graduated Attention

Not all sections of a paper are equally important for reproducibility. This phase applies graduated attention:

- **Deep extraction (line-by-line):** Data description, data preprocessing, methodology, experimental setup, results
- **Summary extraction:** Abstract, introduction, related work, discussion, conclusion
- **Metadata extraction:** Title, authors, publication venue, year, DOI, supplementary material links

This is not a rigid template. Papers vary in structure — some merge methodology and results, some have non-standard section names, some include appendices with critical detail. The extraction process must be dynamic and adaptive, detecting sections by content and intent rather than by heading name alone.

---

## Section-by-Section Extraction Guidelines

### Abstract and Introduction

Extract:
- The core research question or hypothesis
- The domain and subfield
- The claimed novelty or contribution
- Any high-level description of data or methods mentioned

Purpose: Provides orientation for all downstream phases. Helps identify what the paper is trying to prove and what "success" looks like.

### Related Work

Extract:
- Any cited methods that the paper directly builds upon or adapts
- Any datasets cited as baselines or comparisons
- Any papers whose methods are used but not fully described in this paper

**Citation Backstop:** If a method used in this paper is described only by reference to another work (e.g., "we apply the method of Smith et al. [12]"), this must be flagged immediately as a **Citation Dependency**. Log:
- The name of the method
- The full citation reference as given in the paper
- The section and approximate location where this dependency was identified
- A clear note that this method's implementation details are not self-contained in this paper

Do not attempt to infer or reconstruct the method from the citation. Flag and move on.

### Data Section

This is a high-priority extraction target. Extract every available detail:

- Name and description of the dataset(s) used
- Source of the data — is it publicly available, proprietary, or custom-generated by the authors?
- If publicly available: any URLs, DOIs, repository names, or access instructions mentioned
- If custom-generated: the full description of how it was collected, from what sources, using what instruments or APIs
- Time range or version of the data (e.g., "tweets from January to March 2020", "v2.1 of the UCI repository")
- Size of the dataset — number of samples, features, classes, time steps, or any other dimensionality information given
- Data format — CSV, JSON, images, audio, time series, graph, etc.
- Any train/validation/test split described, including split ratios and whether splits are stratified or randomized
- Any known biases, limitations, or caveats about the data mentioned by the authors
- Supplementary data availability — are datasets provided as supplementary files, GitHub links, or external repositories?

Flag any ambiguity. If the paper says "we used a large dataset of medical records" without specifying further, flag this as a **Data Ambiguity** with the exact quote.

### Data Preprocessing and Wrangling

This is the highest-priority extraction target. Every single step matters for reproducibility.

Extract:
- All cleaning steps — removal of duplicates, handling of missing values, outlier treatment
- All transformation steps — normalization, standardization, encoding of categorical variables, log transforms, etc.
- All feature engineering steps — derived features, aggregations, time-based features, embeddings
- Any filtering or subsetting of the data — e.g., "we excluded samples with more than 20% missing values"
- Custom preprocessing pipelines or tools built by the authors
- Order of operations — if the sequence of preprocessing steps is specified or can be inferred, preserve it
- Any randomness involved — shuffling, sampling — and whether a seed is specified
- Any preprocessing that is domain-specific and may require external libraries or tools

For every step, note:
- Is it described precisely enough to implement? If not, flag as a **Preprocessing Ambiguity**
- Are there hyperparameters involved (e.g., a threshold for outlier removal)? Extract the exact value if given
- Is the step referenced from another paper? If so, apply the Citation Backstop

### Methodology and Analysis

Extract:
- The full list of methods, models, or algorithms applied
- For each method: the exact configuration described (hyperparameters, architecture details, training settings)
- The order in which methods are applied
- Any baseline methods used for comparison
- Evaluation metrics used and their exact definitions as given by the authors
- Any cross-validation or statistical testing procedure described
- Hardware or software mentioned as part of the method (e.g., "trained on a single A100 GPU for 48 hours")
- Any ablation studies and what they vary

For every method:
- Is it fully described in this paper? If not, apply Citation Backstop
- Are all hyperparameters given? If some are missing, flag as **Hyperparameter Gap**
- Is randomness involved? Note any seeds or stochastic elements

### Results

Extract:
- All quantitative results reported — tables, figures, inline numbers
- The exact metric values for each method/configuration reported
- Confidence intervals, standard deviations, or statistical significance values if given
- Which results are on which dataset splits (train, validation, test)
- Any qualitative results — visualizations described, patterns highlighted
- The primary claim each result is intended to support

For each result:
- Link it back to the method and data configuration that produced it
- Flag if the result cannot be traced to a specific method or data configuration — **Result Traceability Gap**

### Discussion and Conclusion

Extract:
- The authors' interpretation of their results
- Any limitations acknowledged
- Any future work suggested
- The final conclusions drawn

Purpose: This provides the benchmark for Phase 6. The reproducibility report should assess whether the reproduced results support the same conclusions.

## Citation Backstop: Formal Definition

A Citation Backstop is triggered whenever:

1. A method, algorithm, preprocessing step, or dataset is used in the paper but its implementation details are not provided — only a citation is given
2. A result is compared against a baseline whose methodology is not described in this paper

When triggered, the system must:
- Halt extraction of that specific item
- Log a **Citation Dependency Record** containing:
  - Item type (method / dataset / preprocessing step / baseline)
  - Item name or description
  - Exact citation as given (author, year, reference number)
  - Location in paper (section, paragraph)
  - Impact assessment: is this a core component or a peripheral one?
- Continue extraction of all other items

Citation Dependency Records are included in the PKA as a dedicated section and surfaced to the user at the end of Phase 1.

---

## Output: Paper Knowledge Artifact (PKA)

The PKA is a structured document with the following top-level sections:

1. **Metadata** — title, authors, venue, year, DOI, input source
2. **Summary** — abstract, research question, claimed contributions
3. **Data Description** — all extracted data details
4. **Preprocessing Pipeline** — all extracted preprocessing steps in order
5. **Methodology** — all extracted methods, configurations, hyperparameters
6. **Results** — all quantitative and qualitative results, linked to methods
7. **Conclusions** — authors' stated conclusions
8. **Citation Dependency Records** — all flagged citations with impact assessment
9. **Ambiguity Flags** — all data, preprocessing, and hyperparameter gaps identified
10. **Reproducibility Risk Assessment** — a high-level summary of how complete the extracted information is and where the highest risk of failure lies in downstream phases

The PKA should be queryable: any downstream phase should be able to ask a specific question (e.g., "what normalization was applied to feature X?") and get a precise answer or a clear flag that the information is missing.

---

## Quality Criteria

This phase is successful if:
- No information explicitly stated in the paper is missing from the PKA
- Every ambiguity is flagged rather than silently resolved
- Every citation dependency is recorded with enough detail for a human to follow up if needed
- The PKA is sufficient for Phases 2 through 6 to proceed without re-reading the paper
