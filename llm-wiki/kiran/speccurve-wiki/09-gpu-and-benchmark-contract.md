# GPU And Benchmark Contract

AMD MI300X is load-bearing only if it executes the numerical work that produces the robustness surface. A logo, model endpoint, or unrelated benchmark is not enough.

## Load-Bearing Definition

MI300X is load-bearing when:

- Approved specifications are transformed into batched tensor work.
- CPU and GPU run the same data, same specs, and same resampling count.
- GPU output matches CPU output within a stated numerical tolerance.
- Runtime, throughput, device metadata, and warmup policy are logged.
- The robustness surface depends on the GPU-executed result table.

## Allowed GPU Workloads

Start with one operation and make it real:

- Batched regressions.
- Bootstrap confidence intervals.
- Permutation tests.
- Specification sweeps across model/covariate/exclusion variants.

Do not start with every statistical model. Do not run `statsmodels` in Python loops and call it GPU-backed.

## First Prototype Operation

Build the first MI300X prototype around batched bootstrap resampling for effect-size uncertainty across approved specifications.

Why:

- It creates honest parallel work for moderate-sized datasets.
- It supports reviewer-facing uncertainty intervals.
- It is less dependent on a specific claim shape than a permutation test.
- It can still be compared fairly against a CPU implementation on the same specs and resamples.

Permutation tests remain the preferred second operation when the selected claim has a valid exchangeability structure.

## Benchmark Fairness Rules

| Rule | Required evidence |
|---|---|
| Same data | Dataset hash/path in CPU and GPU logs. |
| Same specs | Shared spec batch ID and count. |
| Same resampling count | Same seeds or seed policy recorded. |
| Same numerical outputs within tolerance | Tolerance metric and pass/fail result. |
| Logged hardware | CPU info, GPU device name, ROCm/HIP/PyTorch versions. |
| Logged runtime | Wall-clock method, start/end timestamps, batch size. |
| Warmup policy stated | Warmup included/excluded or reported separately. |
| No hand-edited results | Benchmark JSON generated by code and committed/exported as artifact. |

## Speedup Formula

```text
speedup = cpu_runtime_seconds / gpu_runtime_seconds
```

Also report:

```text
throughput = total_statistical_runs / runtime_seconds
```

Where:

```text
total_statistical_runs = approved_spec_count * resamples_per_spec
```

If a spec has no resampling, count one run for that spec.

## Acceptance Threshold

- Minimum technical proof: GPU path is real, measured, comparable to CPU, and faster for the selected honest batch.
- Headline demo target: at least 10x speedup on the fair benchmark.
- If speedup is below target, the UI may show the measured result, but the submission must not claim an interactive MI300X advantage without explaining the limitation.
- If CPU/GPU outputs do not match within tolerance, the benchmark fails regardless of speed.
